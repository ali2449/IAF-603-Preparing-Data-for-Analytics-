Module 14: Final Project

Learning Objectives

Upon completion of this module, participants will be able to:

  • Scope and define a real-world data preparation problem.
  • Independently apply the full range of SQL (and Spark SQL) skills learned in the course to a complex, messy dataset.
  • Conduct data profiling, cleaning, transformation, and integration in a systematic workflow.
  • Implement data validation checks to ensure the quality and integrity of the final dataset.
  • Document and present the data preparation process and outcomes clearly and professionally.

Introduction to the Final Project

This final module is the culmination of the entire course. Theory and isolated exercises give way to a comprehensive, hands-on project that mirrors a real-world data preparation task. Working in groups, you will take a raw, "dirty" dataset and apply the full lifecycle of data preparation techniques we have covered—from initial profiling and cleaning to complex transformation, integration, and validation. The goal is not just to write code, but to think critically like a data analyst or engineer: identifying problems, devising strategies, implementing solutions, and documenting your work. This project will solidify your skills and serve as a tangible demonstration of your ability to deliver analysis-ready data.

In greater detail, the final project emphasizes practical application in contexts like healthcare informatics or business analytics. For instance, you might work with datasets from public sources, such as Greensboro’s open data portal, to address real issues like public transit efficiency or health trends. By synthesizing SQL basics, advanced features, and Spark SQL from previous modules, you’ll create a robust data pipeline. This hands-on experience prepares you for professional roles, where data preparation often consumes 80% of analytical time, ensuring you can handle messy, large-scale data effectively.

Project Lifecycle Review

Your project should follow the systematic workflow we’ve implicitly developed over the past 13 modules.

  1. Data Selection & Understanding: Choose a suitable dataset and understand its context, fields, and shortcomings. Ask: What was it collected for? What do the columns mean?
  2. Data Profiling: Use SQL queries (COUNT, GROUP BY, DISTINCT, MIN/MAX) to explore row/column counts, data types, NULL values, ranges, and distributions.
  3. Define a Target Schema: Based on profiling, design the final clean schema, applying normalization (1NF-3NF) and considering multiple related tables.
  4. Develop the Cleaning & Transformation Script: Use SQL/Spark SQL with CTEs/Views for cleaning (handle NULLs/duplicates), standardization (text/dates), transformation (derived features, reshaping), and integration (joins).
  5. Data Validation: Write queries to check row counts, NULLs, key integrity, and value ranges post-cleaning.
  6. Documentation and Presentation: Communicate your process in a report and presentation, covering problems, solutions, challenges, and results.

This lifecycle ensures a thorough, reproducible process. In practice, iterations may occur, such as re-profiling after cleaning to catch new issues. For large datasets, integrate Spark SQL for scalability, especially in validation steps on distributed data lakes.

Example Project Scenario: Greensboro Public Transit Analysis

Imagine your group analyzes public transit data for Greensboro, NC, using messy CSV files: bus_routes.csv, bus_stops.csv, and daily_ridership.csv.

  • Profiling: Discover inconsistent date formats in daily_ridership, NULL accessibility in bus_stops, and comma-separated streets in bus_routes.
  • Cleaning/Transformation: Parse dates with CASE, impute NULLs with COALESCE, split streets into normalized tables.
  • Validation: Check row counts match, no NULL dates, valid links between stops and routes.
  • Final Dataset: Integrated tables ready for analysis, e.g., weekend ridership or accessibility on high-traffic routes.

This scenario demonstrates applying course skills to local issues, like improving transit for Greensboro residents. Expand by incorporating Spark SQL for large ridership logs, ensuring scalability.

Sample Data (bus_routes.csv):

route_id route_name major_streets
1 Route A Main St, Elm St, Oak Ave
2 Route B Pine Rd, Maple Blvd
3 Route C Cedar Ln, Birch Way, Willow St

bus_stops.csv:

stop_id route_id stop_name wheelchair_accessible
101 1 Main St Stop Yes
102 1 Elm St Stop NULL
103 2 Pine Rd Stop No

daily_ridership.csv:

ridership_id route_id date riders
1 1 01/10/2025 150
2 1 2025-01-11 200
3 2 02/01/2025 120 
-- Example cleaning script for dates and NULLs
WITH CleanedRidership AS (
    SELECT
        ridership_id,
        route_id,
        CASE
            WHEN date LIKE '%/%' THEN STR_TO_DATE(date, '%m/%d/%Y')
            ELSE STR_TO_DATE(date, '%Y-%m-%d')
        END AS standardized_date,
        riders
    FROM daily_ridership
),
CleanedStops AS (
    SELECT
        stop_id,
        route_id,
        stop_name,
        COALESCE(wheelchair_accessible, 'Unknown') AS wheelchair_accessible
    FROM bus_stops
)
SELECT *
FROM CleanedRidership cr
JOIN CleanedStops cs ON cr.route_id = cs.route_id;

Explanation: This script standardizes dates using CASE and handles NULLs with COALESCE, joining for integrated analysis. In a full project, extend with validation queries like checking for remaining NULLs or duplicate rows.

Expanded Example (Healthcare Dataset Project): Suppose your group analyzes patient lab results from messy files: patients.csv, labs.csv, admissions.csv. Profiling reveals inconsistent units in labs, missing values in admissions, and denormalized patient info. Clean by normalizing units with CASE, imputing misses with COALESCE, and reshaping for 3NF. Validate by comparing row counts and ranges pre/post-cleaning.

patient_id name age
1001 Alice 45
1002 Bob NULL
lab_id patient_id test_name value unit
1 1001 Glucose 120 mg/dL
2 1001 Glucose 6.7 mmol/L 
-- Normalize glucose units to mg/dL
SELECT
    lab_id,
    patient_id,
    test_name,
    CASE
        WHEN unit = 'mmol/L' THEN value * 18
        ELSE value
    END AS standardized_value,
    'mg/dL' AS standardized_unit
FROM labs
WHERE test_name = 'Glucose';

Explanation: This transformation standardizes units, ensuring comparable analysis. Validate by checking min/max values align with expected ranges post-transformation.

Final Deliverables and Course Review

The final project concludes the course, requiring synthesis of all learned skills.

  • Deliverables:
    1. All documented SQL/Spark SQL scripts.
    2. A detailed project report.
    3. A final presentation.
  • Course Review: The final session includes presentations and a review of key concepts, preparing you for informatics roles.

Deliverables emphasize professional communication, with reports detailing methodology, challenges, and insights. The review reinforces foundational to advanced topics, ensuring comprehensive mastery.

Assessments
