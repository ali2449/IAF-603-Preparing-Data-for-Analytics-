Module 1: Introduction to Data in Informatics

Learning Objectives

Upon completion of this module, participants will be able to:

  • Define and provide concrete examples of structured, semi-structured, and unstructured data.
  • Articulate the primary characteristics, typical sources, and storage methods for each data type.
  • Analyze the implications of different data types on data preparation, analysis, and tool selection.
  • Understand the critical role of data quality dimensions (accuracy, completeness, consistency, timeliness, validity) in informatics.

Introduction to Data in Informatics

Informatics, at its core, is the science of information. Whether in healthcare, business, or scientific research, the ability to extract meaningful insights hinges entirely on the quality and organization of the underlying data. This module serves as the foundational cornerstone of our course, introducing the diverse forms data can take. From the rigid order of a spreadsheet to the free-flowing narrative of a clinical note, understanding these data types is the first critical step in mastering data preparation. We will explore their historical evolution, their prevalence in modern systems, and the unique challenges each presents, setting the stage for the SQL and Spark techniques we will learn to tame them.

Historically, structured data dominated early computing eras with punch cards and relational databases in the 1970s. The rise of the internet in the 1990s introduced semi-structured formats like XML, while the social media boom in the 2000s exploded unstructured data. Today, in fields like informatics, blending these types is common—e.g., combining patient records (structured) with doctor notes (unstructured) and wearable data (semi-structured) for holistic analysis.

The Data Landscape

The digital age has led to an explosion in data volume, velocity, and variety (the "3 Vs" of Big Data). This necessitates a clear understanding of data classification to effectively manage and analyze it. For instance, in a smart city project in Greensboro, NC, traffic sensors provide structured data, social media complaints offer unstructured insights, and API feeds deliver semi-structured updates.

Structured Data

Structured data is the most traditional and organized form of data. It conforms to a predefined data model, making it highly suitable for relational databases.

  • Characteristics:
    • Fixed Schema: Data is organized into tables with rows and columns, where each column has a specific data type (e.g., integer, string, date).
    • Relational: Relationships between tables are defined using keys.
    • Easily Queryable: Highly efficient for querying and analysis using SQL.
  • Typical Sources:
    • Relational Database Management Systems (RDBMS): PostgreSQL, MySQL, SQL Server, Oracle.
    • Enterprise Resource Planning (ERP) systems, Customer Relationship Management (CRM) systems.
    • Financial transaction systems, online forms, sensor data (when pre-processed).
  • Storage: Primarily in relational databases, data warehouses, or simple flat files (CSV, Excel).
  • Implications for Preparation: Generally requires less cleaning for structural issues, but focuses on data quality within the defined schema (e.g., correct data types, valid ranges, no duplicates).
  • How to Recognize Structured Data: Structured data often originates from sources with predefined fields, such as old paper forms (e.g., patient registration forms, survey questionnaires) that are digitized. If the data fits neatly into rows and columns with consistent types—like a spreadsheet where every row has the same number of fields and each field is of a uniform type (e.g., all ages are numbers)—it's structured. For example, a paper census form with fixed blanks for name, age, address can be scanned and converted directly into a database table.
  • Example (Healthcare): A patient demographics table with columns for patient_id, name, date_of_birth, gender, blood_type. Each record fits neatly into these predefined fields.
  • Example (Survey): A survey where respondents select answers from multiple-choice options, resulting in a table with respondent_id, question_1_answer, question_2_answer.
  • Expanded Example (E-commerce): An order table with order_id (INT), customer_id (INT), order_date (DATE), total_amount (DECIMAL). This allows easy aggregation like total sales per day.

Example Structured Data Table (Patient Demographics):

patient_id name date_of_birth gender blood_type
1001 Alice Smith 1985-05-15 Female A+
1002 Bob Johnson 1990-03-22 Male O-
1003 Carol White 1978-11-07 Female AB+

This table illustrates how structured data maintains a consistent format, similar to how old paper medical forms with fixed sections (e.g., "Name:", "DOB:", "Gender:") are transcribed into digital rows and columns.

Semi-Structured Data

Semi-structured data does not conform to a rigid, tabular schema but contains tags or markers that organize and describe the data, allowing for some level of structure. It's more flexible than structured data, making it adaptable to evolving data requirements.

  • Characteristics:
    • Flexible Schema: Schema is not fixed; fields can be added or removed easily.
    • Self-Describing: Data elements are often accompanied by tags or attributes that define their meaning.
    • Hierarchical/Nested: Often organized in a tree-like or graph-like structure.
  • Typical Sources:
    • Web data (JSON from APIs, XML from web services).
    • Log files (server logs, application logs).
    • NoSQL databases (document databases like MongoDB).
    • IoT sensor data streams.
  • Storage: NoSQL databases, file systems (HDFS, S3), specialized data formats (Parquet, ORC).
  • Implications for Preparation: Requires parsing and schema inference. Challenges include inconsistent naming conventions, optional fields, and deeply nested structures that need to be flattened.
  • Example (Healthcare): Data from a wearable health device transmitted as JSON:
{
  "patient_id": "P1001",
  "timestamp": "2024-07-28T10:30:00Z",
  "metrics": {
    "heart_rate": 72,
    "steps": 500,
    "sleep_duration_hours": 7.5
  },
  "location": {
    "latitude": 36.07,
    "longitude": -79.79
  }
}

Another patient's data might omit sleep_duration_hours or add blood_oxygen.

  • Example (Survey): An online survey allowing optional comments or attachments, where the response is stored as XML:
<response id="R2001">
  <demographics>
    <age>35</age>
    <gender>Female</gender>
  </demographics>
  <answers>
    <q1>Strongly Agree</q1>
    <q2>Neutral</q2>
    <comment>I found the survey questions very clear.</comment>
  </answers>
</response>

Expanded Example (Weather API): A weather forecast API response in JSON:

{
  "city": "Greensboro",
  "forecast": [
    {"date": "2025-08-06", "temp": 85, "conditions": "Sunny"},
    {"date": "2025-08-07", "temp": 78, "conditions": "Rainy", "wind_speed": 15}
  ]
}

Note how the "forecast" array allows variable fields like "wind_speed" in some entries but not others, showcasing flexibility.

Unstructured Data

Unstructured data is information that does not have a predefined data model or is not organized in a pre-defined manner. It is the most challenging to process but often contains the richest insights.

  • Characteristics:
    • No Schema: Lacks any internal structure or organization.
    • Heterogeneous: Can be a mix of text, images, audio, video.
    • Context-Dependent: Meaning often relies heavily on human interpretation.
  • Typical Sources:
    • Text documents: Clinical notes, emails, reports, open-ended survey responses, social media posts, articles.
    • Multimedia: Images (X-rays, MRIs), audio recordings (patient interviews), video files.
    • Satellite imagery, surveillance footage.
  • Storage: Data lakes, content management systems, specialized databases for multimedia.
  • Implications for Preparation: Requires advanced techniques like Natural Language Processing (NLP), computer vision, or audio analysis to extract features or convert it into a structured or semi-structured format. High potential for noise, ambiguity, and bias.
  • Example (Healthcare): A physician's clinical note: "Patient presented with severe headache and fever. No rash observed. Prescribed Tylenol. Follow-up in 3 days. History of migraines." Extracting "headache," "fever," "Tylenol," and "migraines" requires NLP.
  • Example (Survey): An open-ended comment from a public health survey: "The new clinic hours are terrible for working parents. I wish they had evening appointments in Greensboro." Extracting sentiment, location, and specific issues requires text analysis.
  • Expanded Example (Social Media): A tweet: "Loving the new park in Greensboro! Great for family picnics, but needs more benches. #CityLife" – Sentiment analysis could classify it as positive with suggestions, while topic modeling identifies "urban planning."
  • Expanded Example (Multimedia): An MRI image file or a video of a patient consultation; processing might involve AI to detect anomalies or transcribe speech.

Data Quality Dimensions

Regardless of the data type, the ultimate goal of data preparation is to ensure data quality. Key dimensions include:

  • Accuracy: Is the data correct and free from errors? (e.g., Is the patient's age actually 45, or was it a typo? In a sales dataset, does the total match the sum of items?)
  • Completeness: Is all necessary data present? (e.g., Are there missing `diagnosis` entries in the `ADMISSIONS` table? Or absent responses in a survey leading to biased results?)
  • Consistency: Is the data uniform across systems and over time? (e.g., Is "Influenza" sometimes spelled "Flu"? Or dates formatted as MM/DD vs DD/MM?)
  • Timeliness: Is the data available when needed and up-to-date? (e.g., Are lab results from yesterday available for today's diagnosis? Stock prices should be real-time for trading.)
  • Validity: Does the data conform to defined business rules or formats? (e.g., Is `age` a positive number? Is `gender` 'Male' or 'Female'? Email addresses should match regex patterns.)

Expanded dimension: Relevance – Does the data align with the analysis goal? E.g., including outdated customer data in a current marketing campaign reduces effectiveness.

Assessments

  1. Conceptual Question: You are analyzing patient feedback. Some feedback is from a structured rating scale (1-5), some from a free-text comment box, and some from a voice recording of a call. Classify each type of feedback and describe one specific data preparation challenge for each.
  2. Application Scenario: For a business in Greensboro, NC, describe how structured, semi-structured, and unstructured data might be used to understand customer behavior. Provide a concrete example for each data type.

Expanded assessment example: Classify data from a retail app: Transaction logs (structured), user session JSON (semi-structured), review texts (unstructured). Discuss preparation steps like normalizing transactions, flattening JSON, and sentiment analysis on reviews.
