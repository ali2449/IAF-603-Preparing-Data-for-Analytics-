Module 1: Introduction to Data in Informatics

Learning Objectives

Upon completion of this module, participants will be able to:

  • Define and provide concrete examples of structured, semi-structured, and unstructured data.
  • Articulate the primary characteristics, typical sources, and storage methods for each data type.
  • Analyze the implications of different data types on data preparation, analysis, and tool selection.
  • Understand the critical role of data quality dimensions (accuracy, completeness, consistency, timeliness, validity) in informatics.

Introduction to Data in Informatics

Informatics, at its core, is the science of information, bridging data, technology, and human decision-making. Whether in healthcare, business, or scientific research, the ability to extract meaningful insights hinges on the quality and organization of the underlying data. This module serves as the foundational cornerstone of our course, introducing the diverse forms data can take: structured, semi-structured, and unstructured. From the rigid order of a spreadsheet to the free-flowing narrative of a clinical note, understanding these data types is the first critical step in mastering data preparation. We will explore their historical evolution, their prevalence in modern systems, and the unique challenges each presents, setting the stage for the SQL and Spark SQL techniques we will learn to tame them.

For instance, in a retail business, structured data might track sales transactions, semi-structured data could capture customer reviews in JSON format, and unstructured data might include customer service call recordings. Each type requires distinct preparation strategies to ensure it can be analyzed effectively.

The Data Landscape

The digital age has led to an explosion in data volume, velocity, and variety (the "3 Vs" of Big Data). This necessitates a clear understanding of data classification to effectively manage and analyze it. The volume of data doubles approximately every two years, velocity increases with real-time streams from IoT devices, and variety spans text, images, and more. This module categorizes data to prepare you for handling these challenges.

Structured Data

Structured data is the most traditional and organized form of data, conforming to a predefined data model, making it ideal for relational databases.

  • Characteristics:
    • Fixed Schema: Data is organized into tables with rows and columns, where each column has a specific data type (e.g., integer, string, date).
    • Relational: Relationships between tables are defined using keys.
    • Easily Queryable: Highly efficient for querying and analysis using SQL.
  • Typical Sources:
    • Relational Database Management Systems (RDBMS): PostgreSQL, MySQL, SQL Server, Oracle.
    • Enterprise Resource Planning (ERP) systems, Customer Relationship Management (CRM) systems.
    • Financial transaction systems, online forms, sensor data (when pre-processed).
  • Storage: Primarily in relational databases, data warehouses, or simple flat files (CSV, Excel).
  • Implications for Preparation: Generally requires less cleaning for structural issues but focuses on data quality within the defined schema (e.g., correct data types, valid ranges, no duplicates).
  • Example (Healthcare): A patient demographics table with columns for patient_id, name, date_of_birth, gender, blood_type. Each record fits neatly into these predefined fields.
  • Example (Retail): A sales transaction table with columns for transaction_id, product_id, quantity, price, and sale_date.
-- Example: Structured data for retail sales
CREATE TABLE Sales (
    transaction_id INT PRIMARY KEY,
    product_id INT,
    quantity INT,
    price DECIMAL(10, 2),
    sale_date DATE
);
INSERT INTO Sales VALUES (1, 101, 2, 29.99, '2025-01-15');

Explanation: This SQL creates a structured table for sales data, ensuring each column has a specific data type and a primary key for uniqueness.

Semi-Structured Data

Semi-structured data does not conform to a rigid, tabular schema but contains tags or markers that organize and describe the data, allowing for flexibility. It's ideal for scenarios where data evolves, such as IoT or web APIs.

  • Characteristics:
    • Flexible Schema: Schema is not fixed; fields can be added or removed easily.
    • Self-Describing: Data elements are accompanied by tags or attributes that define their meaning.
    • Hierarchical/Nested: Often organized in a tree-like or graph-like structure.
  • Typical Sources:
    • Web data (JSON from APIs, XML from web services).
    • Log files (server logs, application logs).
    • NoSQL databases (document databases like MongoDB).
    • IoT sensor data streams.
  • Storage: NoSQL databases, file systems (HDFS, S3), specialized data formats (Parquet, ORC).
  • Implications for Preparation: Requires parsing and schema inference. Challenges include inconsistent naming conventions, optional fields, and deeply nested structures that need to be flattened.
  • Example (Healthcare): Data from a wearable health device transmitted as JSON:
{
  "patient_id": "P1001",
  "timestamp": "2024-07-28T10:30:00Z",
  "metrics": {
    "heart_rate": 72,
    "steps": 500,
    "sleep_duration_hours": 7.5
  },
  "location": {
    "latitude": 36.07,
    "longitude": -79.79
  }
}

Another patient's data might omit sleep_duration_hours or add blood_oxygen, requiring flexible parsing.

  • Example (E-commerce): A customer review stored as JSON:
{
  "review_id": "R1234",
  "customer_id": "C567",
  "product_id": "P789",
  "rating": 4,
  "comment": "Great product, fast delivery!",
  "review_date": "2025-02-01"
}

Explanation: JSON allows optional fields like comment, making it semi-structured. Parsing this into a relational table requires handling missing fields and nested structures.

Unstructured Data

Unstructured data lacks a predefined data model and is the most challenging to process but often contains the richest insights, requiring advanced techniques like NLP or computer vision.

  • Characteristics:
    • No Schema: Lacks internal structure or organization.
    • Heterogeneous: Can be a mix of text, images, audio, video.
    • Context-Dependent: Meaning relies heavily on human interpretation.
  • Typical Sources:
    • Text documents: Clinical notes, emails, reports, open-ended survey responses, social media posts, articles.
    • Multimedia: Images (X-rays, MRIs), audio recordings (patient interviews), video files.
    • Satellite imagery, surveillance footage.
  • Storage: Data lakes, content management systems, specialized databases for multimedia.
  • Implications for Preparation: Requires advanced techniques like Natural Language Processing (NLP), computer vision, or audio analysis to extract features or convert into structured/semi-structured formats. High potential for noise, ambiguity, and bias.
  • Example (Healthcare): A physician's clinical note: "Patient presented with severe headache and fever. No rash observed. Prescribed Tylenol. Follow-up in 3 days. History of migraines." Extracting "headache," "fever," "Tylenol," and "migraines" requires NLP.
  • Example (Social Media): A tweet: "Loving the new coffee shop in Greensboro! Great vibe, but parking is tough. #LocalBusiness" Extracting sentiment and location requires text analysis.

Expanded example: An email from a customer: "I ordered a laptop on Jan 10, but it arrived damaged. Please send a replacement to 123 Main St, Greensboro, NC." Extracting order details, sentiment, and address requires NLP tools.

Data Quality Dimensions

Regardless of data type, the ultimate goal of data preparation is to ensure data quality. Key dimensions include:

  • Accuracy: Is the data correct? (e.g., Is the patient's age 45 or a typo like 450?)
  • Completeness: Is all necessary data present? (e.g., Are diagnosis entries missing in the ADMISSIONS table?)
  • Consistency: Is the data uniform? (e.g., Is "Influenza" sometimes recorded as "Flu"?)
  • Timeliness: Is the data up-to-date? (e.g., Are lab results from yesterday available today?)
  • Validity: Does the data conform to rules? (e.g., Is age positive? Is gender 'Male' or 'Female'?)

Expanded example: In a sales database, ensuring price is positive (validity), all orders have a customer_id (completeness), and product names are standardized (consistency) is critical for accurate reporting.

Assessments

  1. Conceptual Question: You are analyzing patient feedback. Some feedback is from a structured rating scale (1-5), some from a free-text comment box, and some from a voice recording of a call. Classify each type of feedback and describe one specific data preparation challenge for each.
  2. Application Scenario: For a business in Greensboro, NC, describe how structured, semi-structured, and unstructured data might be used to understand customer behavior. Provide a concrete example for each data type.
  3. Practical Exercise: Given a CSV file with columns customer_id, purchase_date, product, and comments, classify each column’s data type (structured, semi-structured, unstructured) and propose one preparation task for each.
